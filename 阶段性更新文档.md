## 二、分阶段实施计划

### 阶段一：后端基础 —— 数据接口补齐

**目标**：API 层面完成所有缺失的 CRUD 和统计接口，前端才能接。

#### 2.1.1 共享类型扩展

**文件**：`packages/shared/src/index.ts`

```diff
// OpeningProject 接口新增字段
+ deletedAt?: string;  // 逻辑删除时间，有值即已删除
```

#### 2.1.2 用户管理接口

**文件**：`apps/api/src/store.service.ts` + `apps/api/src/app.controller.ts`

| 路由 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `GET /users` | 查询 | admin, consultant | 用户列表，支持 `?q=` 搜索，返回 UserAccount[] |
| `GET /users/:id` | 查询 | admin, consultant | 用户详情，含关联项目数统计 |
| `PATCH /users/:id` | 编辑 | admin | 管理员编辑任意用户信息 |
| `PATCH /users/me` | 编辑 | customer | 用户自行修改个人信息 |
| `GET /users/:id/projects` | 查询 | admin, consultant | 该用户的全部项目列表 |

`StoreService` 新增方法：
- `listUsers(actor, query)` — 用户列表+搜索
- `getUser(id)` — 用户详情
- `updateUser(actor, id, input)` — 编辑用户
- `getUserProjects(id)` — 用户项目列表

#### 2.1.3 员工管理接口

**文件**：`apps/api/src/store.service.ts` + `apps/api/src/app.controller.ts`

| 路由 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `GET /staff` | 查询 | admin | 员工列表 |
| `POST /staff` | 新增 | admin | 创建员工（邮箱+密码+姓名+角色+职位+手机） |
| `PATCH /staff/:id` | 编辑 | admin | 编辑员工信息 |
| `POST /staff/:id/toggle-active` | 启用/停用 | admin | 切换员工激活状态 |
| `GET /staff/:id/stats` | 查询 | admin | 员工项目跟进统计 |

`StoreService` 新增方法：
- `listStaff(actor)` — 员工列表
- `createStaff(actor, input)` — 新增员工
- `updateStaff(actor, id, input)` — 编辑员工
- `toggleStaffActive(actor, id)` — 启用/停用
- `getStaffStats(id)` — 员工数据统计（项目数、成交额等）

#### 2.1.4 产品管理补全

**文件**：`apps/api/src/app.controller.ts`

| 路由 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `PUT /catalog/:id` | 编辑 | admin | 编辑产品目录项 |
| `DELETE /catalog/:id` | 删除 | admin | 软删除产品目录项（设置 active=false） |

`StoreService` 新增方法：
- `deleteCatalog(actor, id)` — 软删除产品

#### 2.1.5 数据统计接口

**文件**：`apps/api/src/store.service.ts` + `apps/api/src/app.controller.ts`

| 路由 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `GET /stats/overview` | 查询 | admin | 数据看板聚合数据 |
| `GET /stats/export` | 导出 | admin | 统计导出，`?format=excel|pdf|md` |

`StatsOverview` 返回结构：

```typescript
{
  totalUsers: number;           // 注册用户总数
  totalProjects: number;        // 项目总数（不含已删除）
  projectsByStatus: Record<ProjectStatus, number>;  // 各阶段项目数
  totalRevenue: number;         // 总营业额（所有已成交 Lead(LeadStatus='won') 的 quoteAmount 合计）
  totalWon: number;             // 成交线索数（LeadStatus='won' 的数量）
  consultantStats: { id: string; name: string; projectCount: number; wonCount: number; revenue: number }[];  // 顾问业绩排行
  recentProjects: OpeningProject[];  // 最近10个项目
}
```

#### 2.1.6 导出功能扩展

**文件**：`apps/api/src/excel.service.ts`

现有 `generate` 方法扩展为：

```
generate(project, format: 'excel' | 'pdf' | 'md')
```

- `excel`：现有逻辑，保持
- `pdf`：新增，使用 `pdfkit` 生成。风格简洁清晰，页面顶部固定嵌入公司 Logo（图片路径通过环境变量 `COMPANY_LOGO_PATH` 配置）。内容按报告板块顺序：封面 → 能力评估 → 设备报价 → 盈亏平衡 → 甘特图 → 装修 → 证照 → 岗位
- `md`：新增，生成 Markdown 格式报告文本

#### 2.1.7 项目逻辑删除

**文件**：`apps/api/src/app.controller.ts`

| 路由 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `DELETE /projects/:id` | 删除 | customer, admin | 逻辑删除项目（设置 deletedAt） |

`StoreService` 修改：
- `softDeleteProject(id, actor)` — 设置 `deletedAt = new Date().toISOString()`
- `list()` 修改 — 默认过滤 `deletedAt != null` 的项目；支持 `?status=deleted` 参数查已删除项目
- `get()` 修改 — 不对已删除项目做限制（管理员和销售可查看详情）

## 五、关键设计决策（已确认）

> 前提：该平台仅作为引流获客平台，暂不涉及任何支付模块。目的引流给销售电话/微信沟通客户。

| # | 决策 | 实现方案 |
|---|------|---------|
| 1 | **报价单管理**：顾问在前端直接新增产品行（名称、单价、数量），纯前端操作不落库。导出报表给客户。管理员每月审核顾问新增的产品条目，决定是否正式录入数据库。 | 顾问端报价编辑区改为前端临时行（不调 API），导出功能独立；管理员侧新增"待审核产品"入口 |
| 2 | **销量统计**：按订单状态变为"成交"(LeadStatus='won') 后的 `quoteAmount` 汇总。 | 统计接口取 `leads WHERE status='won'` 聚合 |
| 3 | **逻辑删除**：项目加 `deletedAt` 字段。管理员和销售可通过筛选条件查看已删除项目（不设独立回收站，用列表过滤即可）。 | `list()` 默认过滤 `deletedAt IS NOT NULL`；传 `?status=deleted` 则只查已删除 |
| 4 | **PDF 导出**：简洁风格，含公司 Logo，使用 `pdfkit` 生成。 | 安装 `pdfkit`，在 `excel.service.ts` 中新增 `generatePdf()` 方法，头部固定嵌入 logo 图片 |
